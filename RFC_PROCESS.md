# RFC Process

**Status:** Draft  
**Version:** 0.1.0 Foundation  
**Last Updated:** 2026-07-09  

## Purpose

RFC means Request for Comments.

An RFC is used when a new idea may affect UVDL Core, terminology, architecture, tokens, components, patterns, or tooling.

## When to Create an RFC

Create an RFC for:

- new Core concepts;
- new terminology;
- new pattern families;
- breaking changes;
- tooling proposals;
- changes to governance;
- major implementation strategies.

Do not create an RFC for:

- typo fixes;
- small documentation edits;
- examples;
- non-breaking clarifications.

## RFC Lifecycle

```text
Draft
  ↓
Discussion
  ↓
Prototype
  ↓
Experiment
  ↓
Review
  ↓
Accepted / Rejected / Superseded
```

```markdown
**Status:** Draft  
**Created:** YYYY-MM-DD  
**Related:**  

## Problem

## Hypothesis

## Proposal

## Alternatives

## Risks

## Validation Plan

## Success Metrics

## Decision
```

## Acceptance Criteria

An RFC can be accepted only if it clearly explains:

- the problem;
- the proposed solution;
- alternatives;
- risks;
- validation approach;
- expected practical value.
