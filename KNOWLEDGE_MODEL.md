# Knowledge Model

**Status:** Draft  
**Version:** 0.1.0 Foundation  
**Last Updated:** 2026-07-09  

## Purpose

This document defines how an idea becomes accepted UVDL knowledge.

## Core Principle

UVDL does not treat opinion as knowledge.

A design idea becomes knowledge only after it is documented, reviewed, and validated.

## Knowledge Flow

```text
Observation
  ↓
Problem
  ↓
Hypothesis
  ↓
RFC
  ↓
Prototype
  ↓
Experiment
  ↓
Metrics
  ↓
Analysis
  ↓
Knowledge
  ↓
Core
```

## Knowledge Types

### Observation
A noticed problem, pattern, or opportunity.

### Hypothesis
A proposed explanation or solution.

### Experiment
A practical test on a real or simulated project.

### Metric
A measurable signal used to evaluate the result.

### Knowledge
A validated conclusion that can be reused.

### Core Standard
A stable UVDL rule, pattern, token, or component contract.

## Validation Metrics

Possible metrics include:

- conversion rate;
- task completion rate;
- decision time;
- bounce rate;
- interaction cost;
- accessibility score;
- performance score;
- rework reduction;
- implementation time.

## Knowledge Rule

A claim must not be promoted to Core only because it sounds correct.

It must either be validated in practice or be clearly marked as an engineering assumption.

## First Validation Context

`mechaniki.ru` is the first planned validation context for UVDL decisions.
